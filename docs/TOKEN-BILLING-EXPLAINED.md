# Claude API Token Billing Explained

## Worked example: a "test" request

```
Token Breakdown:
  Input Tokens          3       ← newly sent tokens (placeholder, may be imprecise)
  Output Tokens        44       ← tokens generated by the model
  Cache Creation       229      ← tokens newly written to the cache in this request
    Ephemeral 1h       229      ← 1-hour cache (2x rate)
    Ephemeral 5m         0      ← 5-minute cache (1.25x rate)
  Cache Read       368,518      ← tokens read from the cache (0.1x rate)

Total Context      368,794      ← the full context the model processed
Effective (billed)      47      ← input(3) + output(44) = 47 (cache excluded)
Cache Efficiency     99.9%      ← cache_read / (input + cache_creation + cache_read)
```

## Core concepts

### What is "context"?

When a request is sent to the Claude API, the **entire conversation history** is sent every time:

```
[system prompt] + [full prior conversation] + [new message] → Claude → [response]
```

Sending the 10th request of a Claude Code session means the 1st-9th turns of history plus tool definitions plus the system prompt are all included. That total is the "Total Context".

### Why send the whole thing every time?

The Claude API is **stateless**. The server keeps no conversation state. Every request must carry the full context so the model can understand the prior dialogue.

```
Request 1: [system + tools + "test"]                       → response A (small context)
Request 2: [system + tools + "test" + A + "next question"] → response B (larger context)
Request 3: [system + tools + full history + "another"]     → response C (larger still)
```

## The cache system

### Cache Write (cache creation)

When the first request sends the system prompt, tool definitions, and conversation history, the Anthropic server **stores them in its cache**.

```
First request: [system(100k) + tools(50k) + history(200k) + new(3)]
               ──────────────── stored in cache ────────────────  ─new─
               cache_creation = 350,000 tokens (billed at 2x)
```

### Cache Read

When a later request sends the same context again, the server finds the matching prefix in its cache and **reuses it**:

```
Second request: [system(100k) + tools(50k) + history(200k) + new_history(1k) + new(3)]
                ─────────── read from cache (cache_read) ──────  ───── processed fresh ─────
                cache_read = 350,000 (0.1x rate)
                cache_creation = 229 (the newly added portion)
                input = 3 (the new message)
```

### Why is Cache Read so cheap?

| Token type | What the server does | Cost multiplier |
|-----------|----------------------|----------------|
| **Input** | Full encoding + attention compute | 1x |
| **Cache Creation** | Encoding + attention + cache store | 1.25x–2x |
| **Cache Read** | Load from cache (encoding skipped) | **0.1x** |
| **Output** | Token generation (most expensive) | 5x (vs. input) |

Cache Read only loads already-processed results from the cache, so it requires almost no server GPU compute. That is why it costs **one tenth** of input.

## Relationship with Usage Limit

Anthropic's usage limits (Max plan etc.) are computed on **actual billed cost**:

```
billed cost = input × $5/MTok
            + output × $25/MTok
            + cache_creation_1h × $10/MTok
            + cache_creation_5m × $6.25/MTok
            + cache_read × $0.50/MTok      ← very cheap!
```

### Actual cost of the "test" request (claude-opus-4-6)

```
Input:            3 × $5.00/M     = $0.000015   ≈ $0.000
Output:          44 × $25.00/M    = $0.001100   ≈ $0.001
Cache Create 1h: 229 × $10.00/M   = $0.002290   ≈ $0.002
Cache Read:  368,518 × $0.50/M    = $0.184259   ≈ $0.184
────────────────────────────────────────────────
Total:                              $0.187664   ≈ $0.188
```

If 368,518 tokens were sent without any cache:
```
368,518 × $5.00/M = $1.84 (input alone!)
```

Thanks to the cache, **$1.84 → $0.184** (90% savings)

## What is "Effective (billed basis)"?

In Lupen, "Effective (billed basis) = 47" means:

```
Effective = input(3) + output(44) = 47
```

This represents only the **freshly processed tokens, excluding cache**. Cache tokens were already processed and billed in earlier requests, so this number is what to look at to see "how much genuinely new work this request did".

Note that the actual **cost** still includes cache_read (0.1x but not zero).
